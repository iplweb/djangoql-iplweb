# Синтаксис поиска DjangoQL

## Условия поиска

Условие поиска — это базовый строительный блок поискового запроса. Оно всегда состоит из
3 элементов: `field` (поле), `comparison operator` (оператор сравнения) и `value` (значение),
расположенных именно в таком порядке слева направо.

Пример — поиск пользователей с именем «John». В примере ниже `first_name` — это поле,
`=` — оператор сравнения, а `"John"` — значение:

```
first_name = "John"
```

Ещё один пример — поиск пользователей, зарегистрировавшихся в 2017 году или позже:

```
date_joined >= "2017-01-01"
```

Ещё один пример — поиск суперпользователей:

```
is_superuser = True
```

И ещё один — поиск всех пользователей, чьи имена входят в заданный список:

```
first_name in ("John", "Jack", "Jason")
```

## Несколько условий поиска

Несколько условий поиска можно объединить с помощью логических операторов:
`and` (оба условия должны быть истинными) и `or` (должно быть истинным хотя бы одно условие,
неважно какое). Важно: логические операторы должны быть написаны строчными буквами —
`and` и `or` правильно, а `AND` или `OR` неправильно и вызовет ошибку.

Пример: поиск пользователей с именем «John» `and` зарегистрировавшихся в 2017 году или позже.
Обратите внимание: здесь два условия поиска, соединённых оператором `and`:

```
first_name = "John" and date_joined >= "2017-01-01"
```

Ещё один пример — поиск пользователей, которые являются суперпользователями `or` помечены
флагом «Staff»:

```
is_superuser = True or is_staff = True
```

Логические операторы открывают широкие возможности и позволяют строить сложные поисковые
запросы. При составлении сложного запроса важно помнить одно правило: если запрос содержит
одновременно операторы `and` и `or`, настоятельно рекомендуется использовать скобки для
явного указания приоритета операторов. Следующий пример поясняет, почему это важно.
Предположим, вы хотите найти пользователей, которые являются суперпользователями `or` помечены
флагом Staff, `and` при этом зарегистрировались в 2017 году или позже. Может возникнуть
соблазн написать запрос вот так:

```
is_superuser = True or is_staff = True and date_joined > "2017-01-01"
```

Проблема этого запроса в том, что он сработает не так, как ожидается, поскольку оператор
`and` вычисляется первым. На самом деле он найдёт пользователей, которые либо являются
суперпользователями (независимо от даты регистрации), `or` одновременно имеют флаг Staff `and`
зарегистрировались после 2017 года. Эту проблему решают скобки — просто заключите в них
условия, которые должны вычисляться первыми:

```
(is_superuser = True or is_staff = True) and date_joined > "2017-01-01"
```

Использование скобок рекомендуется только тогда, когда запрос содержит одновременно операторы
`and` и `or`. Если в запросе несколько логических операторов одного вида (только `and` или
только `or`), скобки можно не ставить — всё сработает как ожидается.

## Поля

В поисковом запросе на поля текущей модели следует ссылаться именно так, как они определены
в Python-коде для данной модели Django. Поле ввода поискового запроса оснащено функцией
автодополнения, которая срабатывает автоматически и предлагает все доступные варианты.

В большинстве случаев внутренние поля модели Django выглядят похоже на то, что отображается
в интерфейсе Django admin, только в нижнем регистре и с `_` вместо пробелов. Например, в
стандартном интерфейсе admin для пользователей внутреннее поле `first_name` отображается как
«First name», поле `email` — как «Email address» и т. д. Однако возможны исключения: разработчики
могут задать произвольные отображаемые имена, сильно отличающиеся от внутренних. В таких
случаях имеет смысл попросить разработчиков переопределить этот шаблон справки и разместить
прямо здесь таблицу соответствия «внутреннее имя → отображаемое имя».

Обратите внимание: некоторые поля, видимые в Django admin, могут не поддерживать поиск.
Это касается вычисляемых полей, то есть полей, которые не хранятся в базе данных как простое
значение, а вычисляются на основе других значений в коде.

## Связанные модели

DjangoQL позволяет выполнять поиск по связанным моделям (под капотом связи автоматически
преобразуются в SQL JOIN). Для указания связанных моделей и их полей используйте разделитель
`.` (точку). Например:

```
groups.name in ("Marketing", "Support")
```

Видите `.` в примере выше? Это означает, что `groups` — связанная модель, а `name` — поле
этой модели. Как обычно, автодополнение DjangoQL предлагает варианты для всех доступных
связанных моделей и их полей. Для сложных структур данных можно использовать несколько уровней
связи: сначала указывается связанная модель, затем её связанная модель и так далее.

В большинстве случаев условие поиска со связанной моделью должно указывать конкретное поле
этой модели, а не саму модель. Например, `groups in ("Marketing", "Support")` не сработает,
потому что `groups` — это модель, а не поле. У модели может быть много полей, и сервер не
знает, с каким из них нужно выполнить сравнение. Однако есть одно примечательное исключение:
когда нужно найти записи, которые связаны (или не связаны) с какими-либо связанными моделями
данного типа. В таком случае следует сравнить связанную модель со специальным значением `None`:

```
groups = None
```

Пример выше найдёт пользователей, не входящих ни в одну группу. Чтобы найти всех
пользователей, входящих хотя бы в одну группу, используйте `!= None`:

```
groups != None
```

## Операторы сравнения

| Оператор | Значение | Пример |
| --- | --- | --- |
| `=` | равно | `first_name = "John"` |
| `!=` | не равно | `id != 42` |
| `~` | содержит подстроку | `email ~ "@gmail.com"` |
| `!~` | не содержит подстроку | `username !~ "test"` |
| `startswith` | начинается с подстроки | `last_name startswith "do"` |
| `not startswith` | не начинается с подстроки | `last_name not startswith "do"` |
| `endswith` | заканчивается подстрокой | `last_name endswith "oe"` |
| `not endswith` | не заканчивается подстрокой | `last_name not endswith "oe"` |
| `>` | больше | `date_joined > "2017-02-28"` |
| `>=` | больше или равно | `id >= 9000` |
| `<` | меньше | `id < 9000` |
| `<=` | меньше или равно | `last_login <= "2017-02-28 14:53"` |
| `in` | значение входит в список | `first_name in ("John", "Jack", "Jason")` |
| `not in` | значение не входит в список | `id not in (42, 9000)` |

Примечания:

1. Операторы `~` и `!~` применимы только к полям типа string и date/datetime.
   Поле date/datetime при этом обрабатывается как строковое (например,
   `payment_date ~ "2020-12-01"`).
2. Операторы `startswith`, `not startswith`, `endswith` и `not endswith` применимы
   только к строковым полям.
3. Значения `True`, `False` и `None` можно использовать только с операторами `=` и `!=`.
4. Операторы `in` и `not in` должны быть написаны строчными буквами. `IN` или `NOT IN`
   неправильно и вызовет ошибку.

## Значения

| Тип | Примеры | Комментарии |
| --- | --- | --- |
| string | `"this is a string"` `'another string'` | Строки можно заключать как в двойные кавычки (`"this"`), так и в одинарные (`'this'`). Если строка содержит тот же тип кавычек, что использован для её обрамления, такие символы следует экранировать обратной косой чертой, например `"this is a string with \"quoted\" text"` или `'this is a string with \'quoted\' text'`. |
| int | `42`, `0`, `-9000` | Целые числа — это просто цифры с необязательным унарным минусом. При вводе больших чисел не используйте разделители тысяч: DjangoQL их не понимает. |
| float | `3.14`, `-0.5`, `5.972e24` | Числа с плавающей точкой выглядят как целые числа с необязательной дробной частью, отделённой точкой. Также поддерживается запись через `e` для указания степени десяти. Например, `5.972e24` означает 5,972 × 10²⁴. |
| bool | `True`, `False` | Булев тип принимает только два значения: `True` или `False`. Регистр важен: необходимо писать именно `True` или `False` — с заглавной первой буквой и остальными строчными, без кавычек. |
| date | `"2017-02-28"` | Даты представляются как строки в формате `"YYYY-MM-DD"`. |
| datetime | `"2017-02-28 14:53"` `"2017-02-28 14:53:07"` | Дата и время могут быть представлены как строка в формате `"YYYY-MM-DD HH:MM"` или, опционально, с секундами в формате `"YYYY-MM-DD HH:MM:SS"` (24-часовой формат). Обратите внимание: сравнение с датой и временем выполняется в часовом поясе сервера, как правило UTC. |
| null | `None` | Специальное значение, обозначающее отсутствие какого-либо значения: `None`. Необходимо писать именно так — с заглавной первой буквой и остальными строчными, без кавычек. Используйте его, когда поле в базе данных допускает NULL (в терминах SQL) и нужно найти записи, у которых либо нет значения (`some_field = None`), либо есть хоть какое-нибудь значение (`some_field != None`). |
