# Синтаксис пошуку DjangoQL

## Умови пошуку

Умова пошуку — це базовий будівельний блок пошукового запиту. Вона завжди складається з
3 елементів: `field` (поле), `comparison operator` (оператор порівняння) та `value` (значення),
розташованих саме в цьому порядку зліва направо.

Ось приклад — пошук користувачів з іменем "John". У прикладі нижче `first_name` — це `field`,
`=` — `comparison operator`, а `"John"` — `value`:

```
first_name = "John"
```

Інший приклад — пошук користувачів, які зареєструвалися у 2017 році або пізніше:

```
date_joined >= "2017-01-01"
```

Ще один приклад — пошук суперкористувачів:

```
is_superuser = True
```

І ще один — знаходження всіх користувачів, чиї імена є у заданому списку:

```
first_name in ("John", "Jack", "Jason")
```

## Кілька умов пошуку

Ви можете поєднувати кілька умов пошуку разом за допомогою логічних операторів
`and` (обидві умови мають бути істинними) та `or` (принаймні одна з умов
має бути істинною, байдуже яка). Важливо: логічні оператори мають бути написані
малими літерами: `and` і `or` — правильно, а `AND` або `OR` — неправильно і
спричинять помилку.

Приклад: пошук користувачів з іменем "John" `and` зареєстрованих у 2017 році або
пізніше. Зверніть увагу, що тут є 2 умови пошуку, з'єднані за допомогою `and`:

```
first_name = "John" and date_joined >= "2017-01-01"
```

Ще один приклад — пошук користувачів, які є або суперкористувачами `or` позначені
прапором "Staff":

```
is_superuser = True or is_staff = True
```

Логічні оператори можуть бути дуже потужними, оскільки дозволяють будувати складні
пошукові запити. При побудові складного запиту варто пам'ятати про одну важливу
рекомендацію: якщо ваш запит містить одночасно оператори `and` і `or`, ми наполегливо
рекомендуємо використовувати дужки для визначення пріоритету операторів. Ось приклад,
який ілюструє, чому це важливо. Припустімо, ви хочете отримати список користувачів,
які є або суперкористувачами `or` позначені прапором Staff, `and` зареєстровані у
2017 році або пізніше. Може виникнути спокуса написати такий запит:

```
is_superuser = True or is_staff = True and date_joined > "2017-01-01"
```

Проблема наведеного запиту в тому, що він не робитиме того, що ви очікуєте, оскільки
оператор `and` обчислюється першим. Фактично він поверне користувачів, які є або
суперкористувачами (незалежно від дати реєстрації) `or` одночасно є Staff `and`
зареєстровані після 2017 року. Цю проблему можна вирішити за допомогою дужок —
просто огорніть ними умови пошуку, які мають обчислюватися першими, ось так:

```
(is_superuser = True or is_staff = True) and date_joined > "2017-01-01"
```

Використання дужок рекомендується лише тоді, коли ваш запит поєднує оператори `and` і `or`.
Якщо запит містить кілька логічних операторів лише одного виду (або тільки `and`, або
тільки `or`), дужки можна сміливо опустити — запит виконуватиметься як очікується.

## Поля

У пошуковому запиті слід посилатися на поля поточної моделі точно так, як вони визначені
у Python-коді для відповідної Django-моделі. Поле введення пошукового запиту має функцію
автодоповнення, яка з'являється автоматично і пропонує всі доступні варіанти. Якщо ви
не впевнені, як називається поле, виберіть один із запропонованих варіантів (приклад):

![Приклад автодоповнення DjangoQL](COMPLETION_EXAMPLE_IMG)

У більшості випадків внутрішні поля Django-моделей схожі на те, що ви бачите в
інтерфейсі адміністратора Django, тільки в нижньому регістрі та з `_` замість пробілів.
Наприклад, у стандартному інтерфейсі адміністратора Users внутрішнє поле `first_name`
відображається як `First name`, поле `email` — як `Email address` тощо. Проте можуть
бути винятки: якщо розробники визначили власні назви для відображення, які суттєво
відрізняються від внутрішніх назв. У таких випадках може бути доречним попросити
розробників перевизначити цей шаблон довідки та надати тут відповідність
"внутрішня назва → назва для відображення".

Зверніть увагу, що деякі поля, видимі в адміністраторі Django, можуть бути недоступними
для пошуку. Це стосується обчислюваних полів, тобто полів, які не зберігаються в базі
даних як звичайне значення, а обчислюються з інших значень у коді.

## Пов'язані моделі

DjangoQL дозволяє шукати також за пов'язаними моделями (під капотом автоматично
перетворює зв'язки на SQL-об'єднання). Використовуйте крапку `.` як роздільник для
позначення пов'язаних моделей та їхніх полів. Наприклад:

```
groups.name in ("Marketing", "Support")
```

Бачите `.` у прикладі вище? Це означає, що `groups` — пов'язана модель, а `name` — поле
цієї моделі. Як завжди, автодоповнення DjangoQL пропонує варіанти для всіх доступних
пов'язаних моделей та їхніх полів. Для складних структур даних можна використовувати
кілька рівнів зв'язку: спочатку вказати пов'язану модель, потім її пов'язану модель
і так далі.

У більшості випадків умова пошуку з пов'язаною моделлю має вказувати конкретне поле
цієї моделі, а не саму пов'язану модель. Наприклад, `groups in ("Marketing", "Support")`
не спрацює, оскільки `groups` — це модель, а не поле. Моделі можуть мати багато полів,
і сервер не знає, з яким полем ви хочете виконати порівняння. Проте є один помітний
виняток: коли потрібно знайти записи, пов'язані (або не пов'язані) з будь-якими
пов'язаними моделями такого типу. У такому випадку слід порівняти пов'язану модель
зі спеціальним значенням `None`, ось так:

```
groups = None
```

Наведений вище приклад шукатиме користувачів, які не належать жодній групі. Якщо ж
ви хочете знайти всіх користувачів, які належать принаймні до якоїсь групи,
використовуйте `!= None`:

```
groups != None
```

## Оператори порівняння

| Оператор | Значення | Приклад |
| --- | --- | --- |
| `=` | дорівнює | `first_name = "John"` |
| `!=` | не дорівнює | `id != 42` |
| `~` | містить підрядок | `email ~ "@gmail.com"` |
| `!~` | не містить підрядок | `username !~ "test"` |
| `startswith` | починається з підрядка | `last_name startswith "do"` |
| `not startswith` | не починається з підрядка | `last_name not startswith "do"` |
| `endswith` | закінчується підрядком | `last_name endswith "oe"` |
| `not endswith` | не закінчується підрядком | `last_name not endswith "oe"` |
| `>` | більше | `date_joined > "2017-02-28"` |
| `>=` | більше або дорівнює | `id >= 9000` |
| `<` | менше | `id < 9000` |
| `<=` | менше або дорівнює | `last_login <= "2017-02-28 14:53"` |
| `in` | значення є у списку | `first_name in ("John", "Jack", "Jason")` |
| `not in` | значення відсутнє у списку | `id not in (42, 9000)` |

Примітки:

1. Оператори `~` та `!~` можна застосовувати лише до рядкових та полів дати/дата-час.
   Поле дати/дата-час оброблятиметься як рядкове (наприклад,
   `payment_date ~ "2020-12-01"`)
2. `startswith`, `not startswith`, `endswith` та `not endswith` можна застосовувати
   лише до рядкових полів;
3. Значення `True`, `False` та `None` можна поєднувати лише з `=` та `!=`;
4. Оператори `in` та `not in` мають бути написані малими літерами. `IN` або `NOT IN` —
   неправильно і спричинять помилку.

## Значення

| Тип | Приклади | Коментарі |
| --- | --- | --- |
| string | `"this is a string"` `'another string'` | Рядки можуть бути укладені або у подвійні лапки, як `"this"`, або в одинарні, як `'this'`. Якщо ваш рядок містить той самий тип лапок, що й використовується для обрамлення, такі символи слід екранувати зворотною косою рискою, наприклад `"this is a string with \"quoted\" text"` або `'this is a string with \'quoted\' text'`. |
| int | `42`, `0`, `-9000` | Цілі числа — це просто цифри з необов'язковим унарним мінусом. При введенні великих чисел не використовуйте роздільники тисяч — DjangoQL їх не розуміє. |
| float | `3.14`, `-0.5`, `5.972e24` | Числа з плаваючою крапкою схожі на цілі числа з необов'язковою дробовою частиною, відокремленою крапкою. Також можна використовувати запис `e` для вказівки степеня десяти. Наприклад, `5.972e24` означає 5.972 * 10^24. |
| bool | `True`, `False` | Булевий тип — спеціальний тип, що приймає лише два значення: `True` або `False`. Ці значення чутливі до регістру — слід писати `True` або `False` саме так, з першою літерою у верхньому регістрі та рештою у нижньому, без лапок. |
| date | `"2017-02-28"` | Дати представлені як рядки у форматі `"YYYY-MM-DD"`. |
| datetime | `"2017-02-28 14:53"` `"2017-02-28 14:53:07"` | Дата і час можуть бути представлені як рядок у форматі `"YYYY-MM-DD HH:MM"`, або, за бажанням, із секундами у форматі `"YYYY-MM-DD HH:MM:SS"` (24-годинний формат). Зверніть увагу, що порівняння дати і часу виконується в часовому поясі сервера, який зазвичай є UTC. |
| null | `None` | Це спеціальне значення, що позначає відсутність будь-якого значення: `None`. Його слід писати саме так, з першою літерою у верхньому регістрі та рештою у нижньому, без лапок. Використовуйте його, коли певне поле в базі даних допускає порожні значення (тобто може містити NULL у термінах SQL) і ви хочете знайти записи, які або не мають значення (`some_field = None`), або мають якесь значення (`some_field != None`). |
