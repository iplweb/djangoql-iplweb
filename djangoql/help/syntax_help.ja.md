# DjangoQL 検索構文

## 検索条件

検索条件は、検索クエリの基本的な構成要素です。常に 3 つの要素から成り、左から右の順に `field`（フィールド）、`comparison operator`（比較演算子）、`value`（値）が並びます。

例として、名前が "John" のユーザーを検索してみましょう。以下の例では `first_name` が `field`、`=` が `comparison operator`、`"John"` が `value` です：

```
first_name = "John"
```

別の例として、2017 年以降に登録したユーザーを検索します：

```
date_joined >= "2017-01-01"
```

もう一つの例として、スーパーユーザーを検索します：

```
is_superuser = True
```

さらにもう一つ — 指定したリストに名前が含まれるユーザーをすべて検索します：

```
first_name in ("John", "Jack", "Jason")
```

## 複数の検索条件

複数の検索条件を、論理演算子 `and`（両方の条件が真である必要がある）と `or`（どちらか一方の条件が真であればよい）を使って組み合わせることができます。重要な点として、論理演算子は小文字で記述しなければなりません：`and` と `or` は正しく、`AND` や `OR` は誤りでエラーになります。

例：名前が "John" で `and` 2017 年以降に登録したユーザーを検索します。ここには 2 つの検索条件があり、`and` で結合されていることに注意してください：

```
first_name = "John" and date_joined >= "2017-01-01"
```

もう一つの例として、スーパーユーザー `or` 「スタッフ」フラグが付いているユーザーを検索します：

```
is_superuser = True or is_staff = True
```

論理演算子を使うと複雑な検索クエリを構築できるため、非常に強力です。複雑なクエリを作成する際に重要なポイントがあります：クエリに `and` と `or` の両方の演算子が含まれる場合、演算子の優先順位を明示するために括弧を使うことを強く推奨します。なぜこれが重要なのか、例を挙げて説明します。スーパーユーザー `or` スタッフフラグが付いているユーザーのうち、`and` 2017 年以降に登録したユーザーを取得したいとします。次のようなクエリを書きたくなるかもしれません：

```
is_superuser = True or is_staff = True and date_joined > "2017-01-01"
```

このクエリの問題点は、`and` 演算子が先に評価されるため、意図した結果が得られないことです。実際には、スーパーユーザー（登録日を問わない）`or` スタッフかつ 2017 年以降に登録したユーザーが取得されます。この問題は括弧で解決できます。先に評価すべき検索条件を括弧で囲んでください：

```
(is_superuser = True or is_staff = True) and date_joined > "2017-01-01"
```

括弧を使うのは、`and` と `or` の両方が混在するクエリのときだけ推奨されます。どちらか一種類の論理演算子だけを複数使う場合は、括弧を省略しても意図した通りに動作します。

## フィールド

検索クエリでは、対象 Django モデルの Python コードで定義されているフィールド名をそのまま使用してください。検索クエリの入力欄には自動補完機能があり、自動的にポップアップして利用可能なすべての選択肢を提示します。フィールド名がわからない場合は、表示された選択肢の中から選んでください（例）：

![DjangoQL 補完の例](COMPLETION_EXAMPLE_IMG)

多くの場合、Django モデルの内部フィールド名は Django 管理画面で表示される名前に似ていますが、小文字でスペースの代わりに `_` が使われます。例えば、標準的なユーザー管理画面では、内部フィールド `first_name` は「First name」、`email` フィールドは「Email address」などと表示されます。ただし、開発者がカスタムの表示名を定義している場合は内部名と大きく異なることもあります。そのような場合は、開発者にこのヘルプテンプレートを上書きしてもらい、「内部名 → 表示名」のフィールド対応表をここに記載してもらうとよいでしょう。

なお、Django 管理画面で表示されている一部のフィールドは検索できない場合があります。これには、データベースにそのまま格納されるのではなく、コード内で他の値から計算される「計算フィールド」が含まれます。

## 関連モデル

DjangoQL では関連モデルによる検索も可能です（内部的にリレーションを SQL の JOIN に自動変換します）。関連モデルとそのフィールドを指定するには、`.` ドット区切りを使用します。例：

```
groups.name in ("Marketing", "Support")
```

上記の例の `.` が見えますか？これは `groups` が関連モデルで、`name` がそのモデルのフィールドであることを意味します。通常通り、DjangoQL の自動補完は利用可能なすべての関連モデルとそのフィールドの候補を提示します。複雑なデータ構造に対しては、関連モデル → そのまた関連モデル、という形で複数レベルのリレーションを使用できます。

ほとんどの場合、関連モデルを使った検索条件にはそのモデルの特定フィールドを指定する必要があり、関連モデル自体を指定しても動作しません。例えば、`groups in ("Marketing", "Support")` は動作しません。`groups` はモデルであってフィールドではないからです。モデルには多くのフィールドがあるため、どのフィールドで比較すればよいかサーバーには判断できません。ただし、注目すべき例外が一つあります — 特定の関連モデルに紐付いている（または紐付いていない）レコードを検索したい場合です。その場合は、関連モデルを特別な値 `None` と比較します：

```
groups = None
```

上記の例はどのグループにも属していないユーザーを検索します。逆に、何らかのグループに属しているユーザーをすべて検索したい場合は `!= None` を使います：

```
groups != None
```

## 比較演算子

| 演算子 | 意味 | 例 |
| --- | --- | --- |
| `=` | 等しい | `first_name = "John"` |
| `!=` | 等しくない | `id != 42` |
| `~` | 部分文字列を含む | `email ~ "@gmail.com"` |
| `!~` | 部分文字列を含まない | `username !~ "test"` |
| `startswith` | 部分文字列で始まる | `last_name startswith "do"` |
| `not startswith` | 部分文字列で始まらない | `last_name not startswith "do"` |
| `endswith` | 部分文字列で終わる | `last_name endswith "oe"` |
| `not endswith` | 部分文字列で終わらない | `last_name not endswith "oe"` |
| `>` | より大きい | `date_joined > "2017-02-28"` |
| `>=` | 以上 | `id >= 9000` |
| `<` | より小さい | `id < 9000` |
| `<=` | 以下 | `last_login <= "2017-02-28 14:53"` |
| `in` | 値がリストに含まれる | `first_name in ("John", "Jack", "Jason")` |
| `not in` | 値がリストに含まれない | `id not in (42, 9000)` |

注意：

1. `~` と `!~` 演算子は文字列フィールドおよび日付/日時フィールドにのみ適用できます。日付/日時フィールドは文字列として扱われます（例：`payment_date ~ "2020-12-01"`）
2. `startswith`、`not startswith`、`endswith`、`not endswith` は文字列フィールドにのみ適用できます；
3. `True`、`False`、`None` の値は `=` と `!=` のみと組み合わせられます；
4. `in` と `not in` 演算子は小文字で記述しなければなりません。`IN` や `NOT IN` は誤りでエラーになります。

## 値

| 型 | 例 | コメント |
| --- | --- | --- |
| string | `"this is a string"` `'another string'` | 文字列はダブルクォート（`"this"` のように）またはシングルクォート（`'this'` のように）のどちらでも囲めます。文字列を囲んでいるクォートと同じ種類のクォートが文字列内に含まれる場合は、バックスラッシュでエスケープしてください。例：`"this is a string with \"quoted\" text"` または `'this is a string with \'quoted\' text'`。 |
| int | `42`, `0`, `-9000` | 整数は省略可能な単項マイナス付きの数字です。大きな数値を入力する際に桁区切り文字は使用しないでください。DjangoQL は桁区切り文字を認識できません。 |
| float | `3.14`, `-0.5`, `5.972e24` | 浮動小数点数は、省略可能な小数部をドットで区切った整数のような表記です。10 の累乗を指定するために `e` 表記も使えます。例えば、`5.972e24` は 5.972 × 10^24 を意味します。 |
| bool | `True`, `False` | ブール型は `True` または `False` の 2 つの値のみを受け付ける特殊な型です。これらの値は大文字・小文字を区別します。最初の文字を大文字、残りを小文字にして、クォートなしで `True` または `False` と正確に記述してください。 |
| date | `"2017-02-28"` | 日付は `"YYYY-MM-DD"` 形式の文字列として表現します。 |
| datetime | `"2017-02-28 14:53"` `"2017-02-28 14:53:07"` | 日時は `"YYYY-MM-DD HH:MM"` 形式の文字列、または秒を含む `"YYYY-MM-DD HH:MM:SS"` 形式（24 時間表記）の文字列として表現できます。日時の比較はサーバーのタイムゾーン（通常は UTC）で行われることに注意してください。 |
| null | `None` | これは値が存在しないことを表す特殊な値 `None` です。最初の文字を大文字、残りを小文字にして、クォートなしで正確に記述してください。データベースのフィールドが nullable（SQL では NULL を含み得る）で、値を持たないレコード（`some_field = None`）または何らかの値を持つレコード（`some_field != None`）を検索したい場合に使用します。 |
